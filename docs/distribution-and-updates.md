# 分发与更新建议

## 推荐结构
- 私有源码仓库：保存源码、文案库、GitHub Actions 工作流
- 公共二进制分发点：保存构建出来的 zip、appcast 和更新元数据

## 为什么不推荐直接从私有 GitHub 仓库给最终用户更新
私有 GitHub 仓库对未授权用户不可读。没有 GitHub 账号或没有被邀请成协作者的人，不能直接下载私有仓库内容，也不能稳定地拿它做 app 自动更新源。

## 如果你说“我发出去的人就是有权限的人”
那最简单的做法不是做 app 内鉴权，而是控制分发渠道本身：
- 私有源码放 GitHub
- 构建产物放公开但难猜的下载地址，或者公开 release 仓库
- app 自身只校验更新签名，不做用户登录

## 真正需要鉴权时
如果你要撤回个别人、限制传播、或按用户授权，单纯把 token 塞进 app 里不安全。因为发出去的 app 可以被逆向拿到秘密。真正可控的方案需要你有自己的服务端签发短期令牌。

## 建议的更新路线
1. 源码仍然放私有仓库
2. GitHub Actions 在 macOS runner 上构建 app
3. 构建后把产物发布到公共二进制位置
4. app 用 Sparkle 读取 appcast 做自动更新
5. 更新包用 EdDSA 签名
